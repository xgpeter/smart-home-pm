# PRD 文档模板规范

## Purpose

PRD（产品需求文档）的目标不是"写一份文档"，而是**让研发团队在没有 PM 在场时也能做出正确的产品决策**。好的 PRD 回答三个问题：做什么（What）、为什么（Why）、怎么做（How）。

**何时使用**：新功能立项、版本迭代规划、跨团队需求对齐。

## Key Concepts

### PRD 的黄金法则

| 法则 | 说明 |
|------|------|
| **先说 Why，再说 What，最后 How** | 工程师需要上下文，而不是指令 |
| **每个功能必须有验收标准** | 没有验收标准的需求不是需求，是愿望 |
| **用数据说话** | "用户需要这个"不如"10 位用户中有 8 位在访谈中提到了这个痛点" |
| **标注信心等级** | 对每个需求标注底层假设的信心度%（100%=已验证 50%=竞品参考 20%=直觉） |

### 智能家居 PRD 的特殊性

软件 PRD 可以模糊，错了可以改。硬件 PRD 必须精确，因为硬件决策不可逆：

| 维度 | 软件 PRD | 智能家居硬件 PRD |
|------|---------|-----------------|
| 功能变更成本 | 低（改代码） | 高（重新开模 ¥50K+） |
| 用户验证 | A/B 测试 | 用户访谈 + 众筹 |
| 技术约束 | 服务器资源 | 芯片算力/内存/功耗/认证 |
| 迭代周期 | 1-2 周 | 3-6 个月 |

## 文档结构

> **使用说明**：每个章节标题下的 💡 是该章节的引导性问题，帮助你在填写时思考。如果不确定，标注"待验证"并给出信心百分比。

### 1. 文档信息

💡 谁写的？谁来审？当前版本状态是什么？
- 文档版本号
- 最后更新日期
- 文档作者
- 审核人

### 2. 产品概述

💡 这个产品存在的理由是什么？如果不上线，用户会损失什么？

- 产品背景与目标
- 目标用户群体（≥ 2 个用户画像）
- 核心价值主张（一句话 < 20 字）
- 成功指标（KPI，含测量方式和观察周期）
- 范围边界与非目标（明确说"不做什么"比"做什么"更重要）

### 3. 功能需求

💡 先问：用户的核心 Job 是什么？（参考 `jtbd-and-journey-map.md`）再问：哪些功能是 Must（没有它产品无法使用）？

> 优先级使用 `prioritization-framework.md` 中的 RICE 打分或 Kano 分类。

#### 3.1 核心功能（Must/Should）
- 功能描述 + 该功能解决的用户 Job
- 用户故事（格式：作为/我希望/以便）
- 验收标准（格式：Given/When/Then）
- 优先级 + RICE Score（如有数据）

#### 3.2 增值功能（Could）
- 同上格式
- 标注"本期不做"的 Won't 清单及其原因

### 4. 非功能需求
- 性能要求
- 安全要求
- 兼容性要求
- 可用性要求
- 运维与监控要求

### 5. 用户流程

💡 不要只画快乐路径。用户在哪里卡住？哪里会放弃？参考 `jtbd-and-journey-map.md` 的旅程地图方法。

- 主流程图（从触发到完成）
- 异常处理流程（至少 5 个异常场景）
- 边界情况说明（极端温度、离线、低电量、反锁、老人/小孩使用等）

### 6. 界面原型
- 关键页面描述
- 交互说明
- 视觉规范引用

### 7. 数据需求
- 数据字段定义
- 数据流转说明
- 存储方案

### 8. 技术约束
- 平台限制
- 第三方依赖
- 合规要求

### 9. 项目风险与依赖
- 关键风险清单
- 外部依赖方
- 风险缓解方案

### 10. 数据验证与上线评估
- 埋点/事件定义
- 上线观测指标
- 验收与复盘要求

### 11. 版本规划与里程碑
- MVP 范围
- 后续版本拆分
- 发布时间节点

### 12. 硬件需求规格
- 主控芯片选型（资源预算、功耗预算、备选方案）
- 传感器/外设选型表（型号、接口、关键参数）
- 电源方案（日耗电模型、电池选型、电源树）
- 射频设计要求（天线类型、净空区、匹配网络）
- 机械结构约束（PCB 尺寸、安装孔、连接器、环境适应性）
- BOM 成本框架（分类估算、目标成本）
- 认证要求（SRRC/CCC/电池 UN38.3）
- > 详细模板参见 [hardware-spec-template.md](hardware-spec-template.md)

### 13. 通信协议设计
- 物模型定义（属性/事件/服务三层结构）
- MQTT Topic 规范（命名规则、QoS 策略、遗嘱消息）
- Payload 格式（JSON 通用信封、各消息类型示例）
- 通信策略（心跳机制、上报策略、离线策略）
- 安全设计（设备认证、敏感操作保护、密钥管理）
- > 详细指南参见 [communication-protocol-guide.md](communication-protocol-guide.md)

### 14. 固件架构设计
- RTOS 选型与理由（FreeRTOS / Zephyr / 裸机）
- Flash 分区与内存布局
- 任务划分与优先级（线程数、栈大小、通信机制）
- 核心状态机设计（设备全局、配网、关键业务状态机）
- 低功耗策略（休眠模式、唤醒源、外设电源管理）
- OTA 升级方案（全量/差分、双分区、回滚机制、安全校验）
- 日志与远程诊断（分级存储上报策略）
- 安全启动（签名校验链、防攻击措施）
- 工厂产测（进入方式、测试项、合格标准）
- > 详细模板参见 [firmware-architecture-template.md](firmware-architecture-template.md)

### 15. 测试策略
- 硬件可靠性测试（高低温、跌落、ESD、老化）
- 固件测试（单元测试、集成测试、压力测试）
- App 兼容性测试（主流机型、系统版本）
- 通信稳定性测试（丢包率、断网恢复、并发连接）
- 安全渗透测试

### 16. 量产与售后
- 工厂产测方案
- 包装与配件清单
- 售后诊断系统（远程日志、设备自检）
- 退换货流程

## 编写规范

### 用户故事格式
```
作为 [用户类型]，
我希望 [功能描述]，
以便 [价值/目的]
```

### 验收标准格式
```
Given [前置条件]
When [操作行为]
Then [预期结果]
```

### 优先级定义
- **Must**：必须实现，否则产品无法发布
- **Should**：应该实现，对产品有重要价值
- **Could**：可以实现，有额外价值
- **Won't**：暂不考虑，记录备查

### 风险描述格式
```text
风险项：
影响范围：
触发条件：
缓解方案：
责任人：
```

### 指标定义建议
- 至少包含业务指标、体验指标、稳定性指标各 1 项
- 对每个指标补充统计口径、数据来源和观察周期

## Common Pitfalls

1. **在 PRD 中写实现方案** — PRD 应该描述"做什么"和"为什么"，不是"怎么实现"。把架构决策留给工程师。
2. **验收标准不可验证** — "性能好"不是验收标准。"指纹识别 ≤ 1.0s"才是。每个验收标准必须是可以客观判断真/假的陈述。
3. **缺少非目标清单** — 不说"不做什么"，利益相关方会默认"都要做"。每份 PRD 必须有 Won't 清单。
4. **忽略硬件不可逆性** — 软件可以改，硬件不行。硬件相关的功能决策要多花 3 倍时间验证。参考 `discovery-validation-guide.md`。
5. **用户画像不是真人** — "25-40岁注重生活品质"是人口统计，不是用户画像。用户画像应该有具体的场景、痛点和原话引用。
6. **功能列表代替优先级** — 把所有功能列出来不是 PRD，是愿望清单。必须有取舍，参考 `prioritization-framework.md`。
7. **写完不评审** — PRD 写完后必须做一次跨职能评审（硬件+固件+App+测试+运营），用 `product-checklist.md` 做健康度检查。
8. **忽略认证前置期** — 含无线/生物识别的功能，认证需提前 2-3 个月启动。不要在 PRD 定稿后才想起认证。参考 `certification-guide.md`。
