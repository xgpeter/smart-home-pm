# 智能家居优先级决策框架

## Purpose

智能家居产品涉及硬件、固件、App、云端四条线并行开发，资源冲突频繁，决策失误成本极高（硬件改版 = 重新开模 + 重新认证）。本框架帮助 PM 在有限资源下做出可解释、可复盘的功能优先级决策。

**何时使用**：版本规划、需求评审会、资源冲突时、向管理层汇报优先级依据。

## Key Concepts

### 硬件产品的优先级特殊性

| 软件/SaaS | 智能家居硬件 |
|-----------|-------------|
| 错了可以 hotfix | 错了要召回或重开模 |
| 可以灰度发布 | 硬件功能一次性交付 |
| A/B 测试成本低 | 硬件功能无法 A/B 测试 |
| 迭代周期 1-2 周 | 迭代周期 3-6 个月 |
| 主要成本 = 工程师时间 | 主要成本 = 开模 + 认证 + 备料 |

### 三种优先级框架及适用场景

| 框架 | 适合 | 不适合 |
|------|------|--------|
| **RICE** (Reach×Impact×Confidence÷Effort) | 功能间横向对比，有数据支撑 | 全新品类（Reach 无法估算） |
| **Kano 模型** | 区分基本型/期望型/兴奋型需求 | 技术驱动的功能 |
| **MoSCoW** (Must/Should/Could/Won't) | 版本范围界定，与干系人沟通 | 同优先级内的排序 |

## Application

### Step 1: 确定使用哪个框架

```
决策树：
产品处于什么阶段？
├── 0→1 新品类 → Kano 模型（还不知道用户要什么）
├── 1→N 增长期 → RICE（有数据，需要比较 ROI）
└── 版本边界沟通 → MoSCoW（向管理层/客户承诺范围）
```

### Step 2: RICE 打分（有数据时）

| 维度 | 定义 | 智能家居示例 |
|------|------|-------------|
| **Reach**（覆盖） | 多少用户会用到？ | 指纹开锁 = 100% 用户；猫眼对讲 = 40% 用户（估算） |
| **Impact**（影响） | 对核心指标提升多少？ | 3=巨大(开锁成功率+20%), 2=大, 1=中, 0.5=小, 0.25=微小 |
| **Confidence**（信心） | 数据支撑程度？ | 100%=有 A/B 数据, 80%=有用户研究, 50%=有竞品参考, 20%=直觉 |
| **Effort**（投入） | 人月？ | 通常 0.5-12 人月；硬件需加上开模时间 |

**计算公式：** `RICE Score = (Reach × Impact × Confidence) / Effort`

**打分示例（智能门锁 V1）：**

| 功能 | Reach | Impact | Confidence | Effort(人月) | RICE | 决策 |
|------|-------|--------|------------|-------------|------|------|
| 指纹开锁 | 100% | 3 | 100% | 2 | 150 | ✅ Must |
| 3D 人脸识别 | 60% | 2.5 | 80% | 4 | 30 | ✅ Should |
| 猫眼可视对讲 | 40% | 2 | 70% | 3 | 18.7 | Should |
| HomeKit 接入 | 25% | 1.5 | 60% | 2 | 11.3 | Could |
| 语音助手集成 | 15% | 1 | 50% | 1.5 | 5 | Won't(本期) |

### Step 3: Kano 模型分类（新品类时）

```
从用户访谈中归类每个功能：

基本型需求（Must-be）— 没有它用户会愤怒
  "指纹锁当然要能指纹开锁，不然叫什么指纹锁"
  → 指纹开锁、密码开锁、低电量提醒

期望型需求（Performance）— 越多越好
  "开锁速度越快越好"
  → 开锁速度、电池续航、App 响应速度

兴奋型需求（Attractive）— 没想到但很喜欢
  "开门后灯自动亮了，窗帘也开了，好神奇"
  → 回家场景联动、HomeKey 碰一碰开锁

无差别需求（Indifferent）— 有没有无所谓
  "门锁上能看天气？……也行吧"
  → 非核心功能堆砌
```

**Kano 分类操作步骤：**
1. 列出所有候选功能
2. 对每个功能问用户两个问题：如果有这个功能？（喜欢/理所当然/无所谓/勉强接受/不喜欢）；如果没有？（同上）
3. 用 Kano 评价表交叉得出分类
4. 优先级：基本型 > 期望型 > 兴奋型 > 无差别

### Step 4: 硬件特有的优先级考量

除了框架打分，智能家居硬件还需要额外考量：

| 考量因素 | 说明 | 高风险标志 |
|----------|------|-----------|
| **认证前置** | 某些功能影响认证周期 | 人脸识别 → CCC 生物识别模块检测 +6周 |
| **供应链锁定** | 某些功能依赖特定供应商 | 独家供应的指纹模组，交期 12 周 |
| **硬件耦合度** | 软件可回滚，硬件不可逆 | PCB 改版 → 重新打样 + 重新过 EMC |
| **售后成本** | 复杂功能增加故障率和客服压力 | 自动上锁机构故障率 2% → ¥200/台售后成本 |

## Examples

### 案例：某智能门锁 V1 功能取舍

**背景**：团队想做人脸识别+猫眼+HomeKit，但资源只够做 2 个。

**RICE 分析过程：**

人脸识别：Reach 60% × Impact 2.5 × Confidence 80% / Effort 4 = 30
猫眼对讲：Reach 40% × Impact 2 × Confidence 70% / Effort 3 = 18.7
HomeKit：Reach 25% × Impact 1.5 × Confidence 60% / Effort 2 = 11.3

→ 选人脸识别+猫眼，HomeKit 推迟到 V1.2

**但加入硬件考量后：**

人脸识别 → 需 CCC 生物识别认证 +6周，但人脸模组供应商稳定，交期可控 → ✅
猫眼 → 摄像头模组交期不稳定，且增加结构复杂度（需要前面板开窗）→ ⚠️

**最终决策**：人脸 V1.1 + HomeKit V1.2（因为 HomeKit 是纯软件集成，不依赖硬件变更）。猫眼推迟到结构改版时一起做。

### 失败案例：功能堆砌导致的优先级灾难

某智能音箱团队在 V1 同时塞入：语音助手 + 屏幕 + 摄像头 + Zigbee 网关 + 温湿度传感器。结果：
- 5 条线并行 → 硬件 PCB 改了 4 版
- Zigbee 网关认证未通过 → 延期 3 个月
- 上市时竞品已迭代两代
- **教训**：V1 应该是单点极致（语音+屏幕），其他功能通过 OTA/软件迭代

## Common Pitfalls

1. **把"竞品有"当作优先级依据** — 竞品的人脸识别可能用得很少。看数据，别看功能列表。
2. **Confidence 打分过高** — "老板觉得这个方向对"不等于 Confidence 80%。没有用户验证的数据，最高给 50%。
3. **忽略硬件耦合** — 软件 PM 习惯"先做再说不行就改"，硬件 PM 必须考虑"做了就不能改"的约束。
4. **把 MoSCoW 当排序工具** — MoSCoW 告诉你 Must 比 Should 重要，但不能告诉你两个 Must 之间哪个先做。Must 内部用 RICE 再排。
5. **低估认证对优先级的影响** — 含生物识别/射频的功能，提前 2-3 个月启动认证。
