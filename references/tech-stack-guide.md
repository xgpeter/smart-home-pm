# 智能家居技术栈选型指南

## Purpose

技术选型是智能家居 PM 必须参与的关键决策——选错协议可能意味着产品与主流生态隔绝，选错芯片可能因功耗过高导致电池寿命缩水 50%。本指南帮助 PM 在理解技术权衡的基础上做出可解释的选型判断。

**何时使用**：新产品立项技术评审、芯片选型决策、评估竞品技术方案。

## Key Concepts

**PM 不需要成为工程师，但必须理解三个核心权衡：**
1. **功耗 vs 性能** — 电池设备选低功耗协议（Zigbee/BLE），插电设备可上 WiFi
2. **覆盖 vs 延迟** — Mesh 网络扩大覆盖但增加延迟
3. **开放性 vs 可靠性** — 开放协议兼容性好，封闭系统更稳定

## Common Pitfalls

1. **只看协议参数不看出货量** — Z-Wave 技术好但中国的出货量远低于 Zigbee，供应链和认证成本差异巨大
2. **忽略网关依赖** — WiFi 设备不需要网关是优势，但路由器带机量有限（通常 < 32 台）
3. **选择"最新最酷"的协议** — Thread/Matter 是趋势，但 2026 年设备兼容性仍有限。新品类谨慎，成熟品类可先行
4. **芯片选型只对比价格不对比 SDK 质量** — 便宜的芯片往往开发文档差、社区小，开发周期翻倍

### BLE Mesh

- **优点**：基于 BLE 广播的 Mesh 网络，无需网关、手机直连、低功耗
- **缺点**：延迟较高（洪泛路由）、带宽有限、不适合大数据传输
- **适用**：智能照明（灯泡分组控制）、传感器网络、商场/办公大面积照明
- **不推荐**：视频传输、实时控制场景

| 国产生态 | 泰凌微/杰理/奉加微 | 泰凌微/橙群/涂鸦 |

## Matter 协议

Matter 是由 CSA 联盟（Connectivity Standards Alliance）主导的智能家居应用层标准，前身为 Project CHIP（Connected Home over IP），旨在实现跨品牌、跨生态的设备互操作。

### 协议层次
- **应用层**：Matter 定义设备类型、数据模型、交互行为
- **传输层**：基于 IP 协议，当前支持 WiFi 和 Thread 两种承载方式
- **配网**：基于 BLE 的 Matter 配对流程，支持二维码扫码配网

### Matter 支持设备类型
| 设备类别 | 典型设备 | Matter 版本 |
|----------|----------|-------------|
| 照明 | 灯泡、开关、调光器 | 1.0 |
| 遮阳 | 窗帘电机、百叶帘 | 1.0 |
| 安防 | 门锁、传感器、摄像头 | 1.0 / 1.2（摄像头） |
| 暖通 | 温控器、空调 | 1.0 |
| 家电 | 冰箱、洗衣机、洗碗机 | 1.1 / 1.2 |
| 网关 | Matter Bridge、Controller | 1.0 |

### 选型建议
- **Matter over WiFi**：适合插电设备（音箱、中控屏），无需网关，依赖路由器
- **Matter over Thread**：适合电池设备（传感器、门锁），需 Thread Border Router
- **Matter Bridge**：用于将现有 Zigbee/Z-Wave 设备接入 Matter 生态
- **注意**：Matter 生态仍在建设中，认证设备数量有限，建议关注 CSA 认证产品库

## 云端架构方案

### 方案 A：自建云
- **适用**：大型企业、数据敏感场景
- **技术栈**：AWS/阿里云 + MQTT Broker + 时序数据库
- **成本**：高（服务器 + 运维）
- **优势**：完全可控、数据安全

### 方案 B：IoT 平台
- **适用**：中小型企业、快速上线
- **推荐平台**：
  - 国内：阿里云 IoT、腾讯云 IoT、华为云 IoT
  - 国际：AWS IoT Core、Azure IoT Hub
- **成本**：按量计费
- **优势**：快速接入、稳定可靠

### 方案 C：混合架构
- **适用**：大多数智能家居产品
- **架构**：设备 → 网关 → 云端 → App
- **本地处理**：场景联动、边缘计算
- **云端处理**：远程控制、数据分析、OTA

## 数据存储方案

### 时序数据（传感器数据）
- **推荐**：InfluxDB、TimescaleDB、阿里云 TSDB
- **存储内容**：温度、湿度、电量、开关状态等
- **保留策略**：自动过期清理

### 关系数据（用户/设备信息）
- **推荐**：PostgreSQL、MySQL
- **存储内容**：用户信息、设备绑定、场景配置

### 非结构化数据（日志/配置）
- **推荐**：MongoDB、Redis
- **存储内容**：操作日志、设备配置缓存

## 安全设计要点

1. **设备认证**：证书 + Token 双重认证
2. **通信加密**：TLS 1.3 全链路加密
3. **固件安全**：签名验证、安全启动
4. **数据隐私**：敏感数据加密存储
5. **访问控制**：RBAC 权限模型
6. **OTA 安全**：差分升级、回滚机制

## 选型决策树

```
第一步：设备供电方式？
├── 插电设备 → WiFi (乐鑫 ESP32 系列) 或 Thread (ESP32-C6)
└── 电池设备 → Zigbee (泰凌微 TLSR8258) 或 BLE (奉加微 PHY6212)

第二步：是否需要多协议？
├── 需要 → 泰凌微 TLSR9218 (Zigbee+Thread+BLE) 或 EFR32MG24
└── 不需要 → 单协议即可

第三步：是否需要 Matter 认证？
├── 是 → ESP32-C6 (WiFi+Thread) / TLSR9218 (Thread) / EFR32MG24
└── 否 → 按第一、二步结果

第四步：成本敏感程度？
├── 极致性价比 (<¥5) → 博通BK7231U / 泰凌微TLSR8251 / 杰理AC69
├── 性价比 (¥5-15) → 乐鑫 ESP32-C3/H2 / 奉加微 PHY6212
├── 品质优先 (¥15-30) → Nordic nRF52840 / SiLabs EFR32
└── 高端旗舰 (>¥30) → EFR32MG24 / nRF5340 / 瑞芯微 RK3566
```
