# 智能家居产品发现与验证指南

## Purpose

智能家居硬件开发投入大（开模费 10-50 万、认证费 5-20 万），在产品定义阶段必须验证需求真实性。本指南提供从问题发现到方案验证的完整方法，帮助 PM 在投入工程资源前降低不确定性。

**何时使用**：新产品立项、进入新品类、现有产品重大改版。

## Key Concepts

### 发现 ≠ 需求收集

| 需求收集（错误做法） | 产品发现（正确做法） |
|---------------------|---------------------|
| 用户说"我要人脸识别门锁" | 追问"你现在的开锁方式有什么不方便？" |
| 直接加功能 | 先理解问题 → 生成假设 → 设计实验 → 验证 |
| 竞品有的我也要有 | 竞品没解决好的才是机会 |

### 硬件产品的假设验证阶梯

```
Level 1: 问题假设 — "用户真的有这个痛点吗？"
    ↓ 验证方法：用户访谈（Mom Test 原则）
Level 2: 方案假设 — "这个方案能解决问题吗？"
    ↓ 验证方法：纸面原型/竞品拆解/技术可行性评估
Level 3: 价值假设 — "用户愿意为此付费吗？"
    ↓ 验证方法：众筹测试/预售页面/烟雾测试
Level 4: 量产假设 — "我们的方案能做出来且有利可图吗？"
    ↓ 验证方法：EVT 工程样机 + BOM 精算
```

> 智能家居硬件的沉没成本分水岭在 Level 2→3 之间。开模之前的验证投入 ¥5K-50K，开模后改设计的成本是 ¥100K-500K。

### PoL 验证实验框架（5 种原型类型）

| 原型类型 | 做什么 | 成本 | 适合验证什么 |
|----------|--------|------|-------------|
| Feasibility（可行性） | 关键模块 demo | ¥5K-20K | 指纹模组暗光识别、电机扭矩是否够 |
| Task-Focused（任务聚焦） | 单任务可交互原型 | ¥2K-10K | 配网流程是否顺畅 |
| Narrative（叙事） | 故事板/视频演示 | ¥1K-5K | 回家场景联动的价值感知 |
| Synthetic Data（合成数据） | 模拟数据验证 | ¥0-2K | 日耗电模型、WiFi 连接频次优化 |
| Vibe-Coded（快速原型） | AI 生成可交互原型 | ¥0-1K | App 功能概念验证 |

## Application

### Step 1: 问题框定（Problem Framing）

使用 MITRE 问题框定画布的简化版：

```
Look Inward（向内看）
├── 这个问题在我们的产品线中为什么优先级高？
├── 我们现有的技术栈能解决吗？
└── 团队对这个领域有多少认知？

Look Outward（向外看）
├── 用户当前怎么解决这个问题的？（替代方案）
├── 竞品在这个问题上做得如何？
├── 技术趋势对这个问题是顺风还是逆风？
└── 市场规模和增速？

Reframe（重新框定）
├── 如果问题比我们想的更小/更大呢？
├── 如果用户根本不在乎呢？
└── 如果不是功能问题而是认知问题呢？
```

### Step 2: 假设生成

格式：`我们相信 [用户群体] 有 [问题]，因为 [证据]。如果我们 [方案]，用户会 [行为改变]，表现为 [可测量指标]。`

示例：
```
我们相信 租房用户 有 频繁交接钥匙的痛点，因为
初步访谈 10 位租客中有 8 位提到"换租客时要换锁/配钥匙很麻烦"。
如果我们 提供临时密码+到期自动失效功能，用户会
使用临时密码代替实体钥匙交接，表现为 临时密码月使用次数 ≥ 3 次/设备。
```

### Step 3: 选择验证方法

| 假设级别 | 推荐验证方法 | 成功标准 | 预算 |
|----------|-------------|----------|------|
| 问题假设 | 10-15 位用户访谈 | ≥ 70% 受访者确认痛点 | ¥0-3K |
| 方案假设 | 纸面原型 + 竞品拆解 | ≥ 60% 受访者选择本方案 | ¥2K-10K |
| 价值假设 | 众筹页面/预售 | 转化率 ≥ 3% 或 NPS ≥ 40 | ¥5K-30K |
| 量产假设 | EVT + BOM精算 | BOM 成本 ≤ 目标 70%，毛利 ≥ 40% | ¥50K-200K |

### Step 4: 用户访谈（Mom Test 原则）

**不要问的问题：**
- ❌ "你会买一个人脸识别门锁吗？"（假设性问题，无意义）
- ❌ "你觉得这个功能怎么样？"（礼貌性回答，不可靠）
- ❌ "如果价格是 ¥1999 你会买吗？"（实际购买行为不可预测）

**应该问的问题：**
- ✅ "你上次忘带钥匙是什么时候？后来怎么解决的？"（挖掘真实行为）
- ✅ "你现在用的门锁有什么不方便的地方？"（发现痛点）
- ✅ "你为这个问题试过什么解决方案？花了多少钱？"（验证付费意愿）

**访谈后记录格式：**
```
用户：#12 张先生，32岁，租房，已用智能门锁 1 年
痛点确认：交接钥匙麻烦 ✓
意外发现：他更担心的是前租客复制了钥匙（安全焦虑 > 便利需求）
引用："我每次换房第一件事就是换锁芯，花 300 块买个安心"
```

### Step 5: 综合与决策

```
验证结果矩阵：
                     强证据          弱证据
高影响（如果为真）  → 立项推进       → 追加验证
低影响（如果为真）  → 可做可不做     → 放弃
```

## Examples

### 成功案例：云鲸自清洁拖布

- **问题发现**：用户访谈发现"洗拖布是最讨厌的环节"，不是"扫地不干净"
- **假设**：如果拖布能自己洗，用户会更频繁使用拖地功能
- **验证**：众筹页面 A/B 测试，"自清洁"版本转化率高 40%
- **结果**：J1 首发当月破亿

### 失败案例：某智能冰箱的"食材管理"功能

- **问题**：PM 假设用户需要"冰箱告诉我里面有什么"
- **实际**：用户访谈发现，用户知道自己冰箱里有什么，需要的是"快过期提醒"和"自动生成购物清单"
- **教训**：跳过了 Mom Test，直接做了摄像头+AI 识别方案，成本高且没解决正确的问题
- **损失**：开模 + AI 算法投入约 200 万，功能使用率 < 5%

## Common Pitfalls

1. **在用户访谈中推销自己的方案** — 用户会礼貌性赞同，但不会真的买。访谈目标是发现真相，不是获得认可。
2. **混淆"竞品有"和"用户需要"** — 竞品的功能可能是错误的，用数据验证而不是跟风。
3. **用技术可行性替代需求验证** — "我们能做到"不等于"用户需要"。
4. **跳过 Level 2 验证直接开模** — 智能家居硬件最大的坑。先用 3D 打印外壳 + 开发板验证，再开模。
5. **把早期用户的热情当作市场规模** — 科技爱好者的反馈不能代表大众市场。
